# План доработки (t1..t17)

Формулировки ориентированы на доработку текущей базы кода (переиспользуем `bots_fonctionnels/bot_maisonRabotaetV1.py` как наиболее близкий к ТЗ). Порядок: сначала тесты, затем реализация.

## t1 — Минимальные защитные тесты ядра функционала
- Результат: быстрый набор тестов, который сразу ловит регрессии при доработках.
- Детали:
  - дедуп/новизна: повторная вставка URL не даёт дубликатов, «новое» только при первом появлении;
  - `normalize_url`: удаляет `utm_*`, `gclid`, `fbclid`, `#fragment`, но сохраняет значимые query‑параметры;
  - каталог vs карточка: `looks_like_catalog` и `looks_like_listing_path` отсекают списки и принимают карточки;
  - `resolve_to_listing`: предпочитает `canonical/og:url` и не возвращает каталог;
  - цена: `extract_price_from_jsonld` и `extract_price_from_text` для базовых кейсов;
  - alive‑check: `200` без маркеров снятия = активна, `404/410` или маркеры снятия = неактивна;
  - CSV: имена `{YYYY-MM-DD}_{HH-MM-SS}_daily_results.csv` и `{...}_history_results.csv` + корректные заголовки;
  - Telegram: формат «источник + заголовок + URL» без дублей (через мок).

## t2 — Уточнить и зафиксировать конфигурацию поиска под дома
- Результат: единый список параметров (тип = `maison`, покупка/аренда, бюджет/комнаты/метраж, города/радиус), которые будут конфигурируемыми.
- Детали:
  - вынести «жёсткие» константы (`BUDGET`, `CITIES_*`, `BASE_FILTERS`, `DOMAINS`) в параметры/конфиг;
  - решить формат: env vars + значения по умолчанию (как сейчас) или отдельный `config.py`.

## t3 — Выделить модель объявления и целевые поля под CSV
- Результат: чётко определённый набор полей (минимум из ТЗ) и правила, когда поле пустое.
- Детали:
  - `date_checked`, `source`, `city`, `title`, `price_eur`, `rooms`, `area_m2`, `url`;
  - определить `source`: брать `domain_of(url)` или `displayLink` из Google (с нормализацией `www.`).

## t4 — Привести генерацию CSV к требованиям ТЗ (только формат/имена)
- Результат: функции записи CSV, которые создают новые файлы с именами:
  - `{YYYY-MM-DD}_{HH-MM-SS}_daily_results.csv`
  - `{YYYY-MM-DD}_{HH-MM-SS}_history_results.csv`
- Детали:
  - единый timestamp для пары файлов в одном запуске;
  - разделитель и кодировка: стандартизировать (рекомендация: `,` + `utf-8` как в примере ТЗ).

## t5 — Расширить/зафиксировать схему SQLite под историю и актуальность
- Результат: SQLite — основной источник истины для «видели/не видели» и «актуально/не актуально».
- Детали:
  - проверить, достаточно ли текущих колонок (`last_checked`, `is_active`) и добавить при необходимости:
    - `last_status_code`, `inactive_reason`, `last_seen` (если нужно);
  - добавить индексы по `is_active`, `last_checked` (для скорости выборок);
  - зафиксировать трактовку:
    - «новое» = `url` отсутствует в БД вообще (даже если потом стало неактуальным).

## t6 — Реализовать проверку актуальности объявления (alive-check)
- Результат: функция `check_listing_active(url) -> (is_active, status_code, reason)` по правилам ТЗ.
- Детали:
  - HTTP: считать активным только при `200 OK`;
  - контент-маркеры снятия (примерный список): `annonce supprimée`, `annonce expirée`, `n’est plus disponible`, `n'est plus disponible`, `vendu`, `plus disponible`;
  - аккуратная обработка редиректов и антибот-страниц (логировать отдельной причиной).

## t7 — Доработать пайплайн кандидатов: normalize → resolve → alive-check → extract
- Результат: на входе URL из Google, на выходе либо «валидное активное объявление», либо отказ с причиной.
- Детали:
  - переиспользовать `resolve_to_listing` из `bot_maisonRabotaetV1.py`;
  - после резолва выполнять `alive-check` уже по конечному URL;
  - если URL неактуален — обновлять БД (`is_active=0`, `last_checked=...`, причина).

## t8 — Извлечение метаданных (city/rooms/area) для CSV
- Результат: максимальное заполнение полей `city`, `rooms`, `area_m2` (если доступны), иначе пусто.
- Детали:
  - источники в порядке приоритета:
    1) JSON-LD (`numberOfRooms`, `floorSize`, `addressLocality` / `address.addressLocality`);
    2) OpenGraph/meta (`og:title`, `og:description`) + regex;
    3) текст страницы + regex (как фоллбек);
  - нормализовать единицы (`m²`/`m2`) и числа (запятая/точка).

## t9 — Обновление БД по результатам проверки и извлечения
- Результат: после каждого запуска в БД есть актуальный статус по всем проверенным URL.
- Детали:
  - для новых URL: вставка + `first_seen`, заполнение метаданных (если извлекли);
  - для существующих: обновление `last_checked`, `is_active`, метаданных (по желанию: только если пустые).

## t10 — Формирование `daily_results.csv` (только новые + только активные)
- Результат: список только тех URL, которые:
  - впервые увидены в этом запуске,
  - прошли alive-check,
  - имеют доступные метаданные (частично).
- Детали:
  - порядок: по времени обнаружения или по домену/цене (определить);
  - лимит отправки в Telegram (`MAX_SEND`) не должен «урезать» запись в CSV (CSV пишет всё; Telegram — первые N).

## t11 — Формирование `history_results.csv` (все активные на момент запуска)
- Результат: файл, который содержит **все** URL, которые считаются активными на момент завершения проверки.
- Детали:
  - стратегия: перед записью `history` пробежать alive-check по всем URL в БД (или по тем, что не проверялись давно), чтобы `is_active` соответствовал моменту запуска;
  - `history` должен включать и новые (из `daily`) и старые активные объявления.

## t12 — Telegram: сделать ссылки гарантированно кликабельными
- Результат: сообщения Telegram в формате «источник + заголовок + кликабельная ссылка», без дублей.
- Детали:
  - выбрать `parse_mode` (`HTML` или `MarkdownV2`) и добавить экранирование;
  - отправлять только новые активные (содержимое `daily`), возможно батчить по 10–20 строк, если сообщение слишком длинное.

## t13 — Логирование и метрики (как в ТЗ)
- Результат: логи по ключевым счётчикам и причинам отбраковки.
- Детали:

  - сколько новых попало в `daily`,
