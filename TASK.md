# Техническое задание

## Задача
Нужна автоматизация для поиска домов в районе Лилля (Франция). Скрипт выполняет поиск по заданным критериям, отправляет новые результаты в Telegram (кликабельные ссылки), а также ведёт два CSV‑файла:
1) **`*_daily_results.csv`** — только **новые** объявления (не встречавшиеся ранее), которые актуальны на дату запуска.
2) **`*_history_results.csv`** — все объявления, которые бот когда‑либо находил и которые актуальны на дату запуска; при каждом запуске добавляет новые объявления, проверяет актуальность старых и исключает неактуальные из итогового файла текущего запуска.

## Функциональные требования

### Поиск
- Источник: Google Custom Search API **или** прямые сайты‑доноры (список уточняется).
- Фильтры: Лилль и пригороды; тип жилья «дом»; цена/площадь/комнаты — конфигурируемые параметры.

### Telegram
- Отправлять только новые объявления, без дублей.
- Формат: источник + заголовок + кликабельная ссылка.

### `daily_results.csv`
- Содержит **только новые** объявления, обнаруженные в текущем запуске (например, 25 января 2026), которых не было в истории ранее.
- «Актуальное» = страница доступна (HTTP 200) и нет признаков снятия (“annonce supprimée”, “annonce expirée”, “n’est plus disponible”, “vendu”).
- Файл **не перезаписывается**: при каждом запуске создаётся **новый** файл с таймстампом.

### `history_results.csv`
- Накопительный список **только актуальных** ссылок на момент запуска (за всю историю поисков бота).
- При каждом запуске:
  - добавлять новые актуальные объявления (из `daily_results.csv` текущего запуска),
  - проверять актуальность всех сохранённых ссылок,
  - исключать неактуальные из итогового файла текущего запуска.
- Файл **не перезаписывается**: при каждом запуске создаётся **новый** файл с таймстампом.

### Именование CSV файлов (важно)
- Все CSV‑файлы создаются **новыми** на каждый запуск.
- Формат имени: `{YYYY-MM-DD}_{HH-MM-SS}_daily_results.csv` и `{YYYY-MM-DD}_{HH-MM-SS}_history_results.csv`.
- Примеры:
  - `2026-01-25_14-07-32_daily_results.csv`
  - `2026-01-25_14-07-32_history_results.csv`

### Хранилище истории (важно)
- Чтобы определять “новая ссылка” и хранить историю между запусками, нужно персистентное хранилище.
- Допустимые варианты:
  - SQLite (рекомендуется): хранить URL и метаданные, а также результат последней проверки актуальности.
  - Либо чтение последнего `*_history_results.csv` как источника “уже известных” URL (менее надёжно).

### Данные в CSV
- Минимально: `date_checked`, `source`, `city`, `title`, `price_eur`, `rooms`, `area_m2`, `url`.
- Если данных нет — поле пустое.

## Пример `daily_results.csv`

```csv
date_checked,source,city,title,price_eur,rooms,area_m2,url
2026-01-25,leboncoin.fr,Lambersart,"Maison de ville 4 pièces avec garage",349000,4,95,https://www.leboncoin.fr/...
2026-01-25,bienici.com,Wasquehal,"Maison 5 pièces avec jardin",289000,5,100,https://www.bienici.com/...
2026-01-25,pap.fr,Mouvaux,"Maison T4 rénovée",339000,4,90,https://www.pap.fr/...
```

## Нефункциональные требования
- Конфигурация через переменные окружения (API‑ключи, Telegram токен, chat ID).
- Ограничение запросов (sleep/лимиты).
- Логирование: сколько найдено кандидатов, сколько новых живых попало в `daily`, сколько всего живых попало в `history`, сколько стало неактуальным.

## Выходные артефакты
- CSV‑файлы с таймстампом при каждом запуске.
- Python‑скрипт(ы), запускаемые командой `python ...`.

## Критерии готовности
- Telegram получает только новые объявления.
- `*_daily_results.csv` содержит **только новые** актуальные объявления текущего запуска.
- `*_history_results.csv` содержит **все** актуальные ссылки на момент запуска (за всю историю), без неактуальных/битых.
- После каждого запуска создаются новые CSV‑файлы с таймстампом; предыдущие остаются.

# Аудит текущего состояния репозитория (что уже есть)

## Уже реализовано
- Есть два работающих бота в `bots_fonctionnels/`: `bot_maison3.py` и `bot_maisonRabotaetV1.py`.
- Используется SQLite `annonces_metropole_lille.sqlite` с таблицей `listings` и `UNIQUE(url)`; уже есть миграция колонок `price_eur`, `city`, `rooms`, `area_m2`, `last_checked`, `is_active`.
- Поиск через Google Custom Search API (CSE), фильтрация по списку доменов (`site:...`), нормализация URL (удаление `utm_*`, `gclid`, `fbclid`).
- Telegram-уведомление отправляется одним сообщением со списком ссылок.
- В `bot_maisonRabotaetV1.py` уже есть:
  - эвристики отличать карточку от каталога (`looks_like_catalog`) и мягкая проверка URL карточки,
  - «дорезолв» ссылок до карточки (canonical/og:url/json-ld/поиск лучшей ссылки в HTML),
  - извлечение цены из JSON-LD/текста.
- Есть минимальные unit-тесты утилит в `tests/test_bot_utils.py`.

## Не покрыто ТЗ / расхождения с ТЗ
- CSV сейчас один и с другим именованием (`nouvelles_annonces_...csv`), и структура колонок не совпадает с ТЗ.
- Нет механики **двух файлов** на каждый запуск: `{timestamp}_daily_results.csv` и `{timestamp}_history_results.csv`.
- Нет обязательной проверки **актуальности объявления** по HTTP + «признаки снятия» (текстовые маркеры).
- Нет гарантии, что `history_results.csv` содержит **только актуальные** объявления на момент запуска: в БД пока не ведётся/не обновляется `is_active` как источник истины.
- Не извлекаются (или не гарантируются) `rooms` и `area_m2`, `city` — для требуемого CSV.
- Telegram-формат в ТЗ: «источник + заголовок + кликабельная ссылка»; сейчас кликабельность зависит от клиента, а экранирование/parse_mode не настроены.

## t1 — Минимальные защитные тесты ядра функционала
- Результат: быстрый набор тестов, который сразу ловит регрессии при доработках.
- Детали:
  - дедуп/новизна: повторная вставка URL не даёт дубликатов, «новое» только при первом появлении;

  - каталог vs карточка: `looks_like_catalog` и `looks_like_listing_path` отсекают списки и принимают карточки;
  - `resolve_to_listing`: предпочитает `canonical/og:url` и не возвращает каталог;
  - цена: `extract_price_from_jsonld` и `extract_price_from_text` для базовых кейсов;

  - CSV: имена `{YYYY-MM-DD}_{HH-MM-SS}_daily_results.csv` и `{...}_history_results.csv` + корректные заголовки;
  - Telegram: формат «источник + заголовок + URL» без дублей (через мок).
