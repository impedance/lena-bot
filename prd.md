
Описание запроса клиента
Клиент хочет автоматизированное решение, которое регулярно находит и отбирает наиболее релевантные объявления о продаже домов в заданном регионе, строго соответствующие его критериям, и предоставляет только качественную, уже отфильтрованную подборку.

🎯 Основная задача
Система должна:
Искать новые объявления о продаже домов в определённых городах
Отбирать только те варианты, которые максимально соответствуют требованиям клиента
Исключать нерелевантные предложения (квартиры, аренду, неподходящие города и т.д.)
Давать клиенту готовый список проверенных вариантов, без «шума»
📌 Важно: клиенту нужен не просто сбор объявлений, а именно интеллектуальная фильтрация и приоритизация результатов

💡 Итоговая ценность для клиента
Такой инструмент решает ключевую проблему:
❌ «Мне приходится вручную просматривать десятки сайтов и сотни объявлений»
✅ «Я получаю уже отфильтрованный список только подходящих вариантов»

🧩 Как это можно сформулировать кратко
Клиентский запрос:
Нужна система, которая автоматически находит новые объявления о продаже домов в заданных районах и отбирает только те, которые максимально соответствуют моим критериям, исключая нерелевантные варианты и предоставляя мне готовую, качественную подборку.

🧩 5. Итоговая логика продукта
Можно сформулировать так:
Система не просто ищет объявления, а действует как «аналитик недвижимости», который:
агрегирует рынок
фильтрует по строгим критериям
отбрасывает шум
выделяет только лучшие варианты
и при этом не теряет потенциальные возможности

🧠 Логика работы (с точки зрения результата)
1. Поиск максимально полного набора вариантов
Система просматривает множество источников, чтобы не упустить ни одного потенциально подходящего объекта.
👉 Цель: высокая полнота (coverage)
2. Жёсткая фильтрация по критериям
Каждое объявление проверяется на соответствие ключевым требованиям:
тип объекта: только дома
количество комнат (≥ 3)
площадь (от заданного минимума)
наличие участка / сада
бюджет (до заданного лимита)
дополнительные параметры (гараж, отопление и т.д.)
география (строго заданные города)
👉 Нерелевантные объявления автоматически исключаются
3. Умная классификация результатов
Объявления делятся на категории:
✅ Полностью подходящие — приоритетные (основной результат)
⚠️ Потенциально подходящие — если данных не хватает (например, нет информации о площади)
❌ Исключённые — если явно не соответствуют
👉 Это позволяет не терять интересные варианты, даже если информация неполная
4. Очистка и проверка качества
Система:
убирает дубликаты
старается привести ссылки к конкретным объявлениям (а не спискам)
помечает сомнительные ссылки, чтобы клиент понимал уровень надёжности
👉 Цель: дать максимально «чистый» и удобный список
5. Приоритизация и доставка результата
Клиент получает:
только лучшие (строго подходящие) объявления
в ограниченном и удобном формате (без перегрузки)
с акцентом на новые предложения
👉 Это экономит время и исключает необходимость ручного поиска

🧠 2. Как определяется статус объявления
✅ Полностью подходящие (OK)
Объявление считается приоритетным, если:
тип = дом
не аренда, не квартира, не коммерция
город из списка разрешённых
площадь ≥ 90 м²
≥ 3 спальни
нет запрещённых ключевых слов
👉 Это «чистые» лиды, которые отправляются клиенту
⚠️ Потенциально подходящие (MAYBE)
Объявление попадает сюда, если:
    1. Не хватает данных:
нет информации о площади
нет информации о количестве комнат
    2. География не определена:
город не указан в тексте
    3. Частично не хватает критериев:
нет упоминания участка
👉 Важно:
Такие объявления не выбрасываются, чтобы не терять потенциально хорошие варианты
❌ Исключённые (EXCLUDED)
Объявление сразу отбрасывается, если:
🚫 Явное несоответствие:
квартира / студия
аренда
коммерческая недвижимость
🚫 Не тот город:
найден запрещённый город
🚫 Параметры ниже минимума:
площадь < 90 м²
< 3 спальни
🚫 Спец-слова:
viager
nue-propriété
plain-pied
flamande
👉 Эти объявления никогда не доходят до пользователя

Ниже — точная логика того, как бот присваивает статус каждому объявлению:
OK / MAYBE_* / EXCLUDED. Это не общая интерпретация, а именно текущая логика из кода

Общий принцип
Для каждого объявления бот собирает единый текстовый контекст из:
URL
заголовка
сниппета
После этого он ищет в этом тексте признаки соответствия или несоответствия. На основе этого возвращается один из статусов:
OK — полностью подходит
MAYBE_* — может подойти, но не хватает подтверждения
EXCLUDED — явно не подходит

4. Важная особенность: порядок проверок
Порядок очень важен, потому что бот возвращает первую сработавшую причину.
Примерно логика такая:
Проверить запрещённые слова
Проверить аренду
Проверить квартиру
Проверить коммерцию
Проверить город
Проверить площадь
Проверить спальни
Проверить участок/сад
Если ничего проблемного не найдено — OK
Это значит, например:
если объект без площади, но это квартира — он будет EXCLUDED, а не MAYBE_SURFACE
если город запрещён, бот не будет дальше разбираться, есть ли сад или спальни

2. Когда объявление становится ⚠️ потенциально подходящим
Потенциально подходящие — это не «почти подошли» в абстрактном смысле, а объявления, где нет явного несоответствия, но нет и достаточного подтверждения, чтобы считать их полностью подходящими.
В коде это выражено через статусы:
MAYBE_CITY
MAYBE_SURFACE
MAYBE_BEDROOMS
MAYBE_TERRAIN
Важно: такие статусы возникают только в режиме STRICT, когда бот требует более жёсткое подтверждение критериев. Во fallback-режиме часть таких случаев может не помечаться как MAYBE столь строго
2.1. MAYBE_CITY — город не подтверждён
Если:
не найден ни один разрешённый город
и при этом не найден запрещённый город
то география считается неизвестной
В режиме STRICT это не исключение, а статус:
MAYBE_CITY
причина: city_unknown
Что это значит по смыслу
Объект может быть в нужной зоне, просто в тексте выдачи это не видно.
Например:
в заголовке нет города
в сниппете город не попал
в URL нет географической подсказки
То есть бот не может подтвердить географию, но и не видит явного нарушения.
2.2. MAYBE_SURFACE — площадь не указана
Если бот вообще не нашёл ни одного значения площади в формате m²/m2, то:
при STRICT объявление получает
MAYBE_SURFACE
причина: surface_missing
Что это значит по смыслу
Бот не знает, больше площадь 90 м² или меньше.
Он не исключает объект, потому что доказательства несоответствия нет.
2.3. MAYBE_BEDROOMS — нет информации о спальнях
Если бот не нашёл ни одного указания на количество спален/комнат в формате X chambre или X ch, то:
при STRICT ставится
MAYBE_BEDROOMS
причина: bedrooms_missing
Что это значит по смыслу
Объект может иметь 3+ спальни, но бот не может это подтвердить по доступному тексту.
2.4. MAYBE_TERRAIN — не найдено упоминание участка/сада
Если в тексте нет ни одного из слов:
jardin
terrain
extérieur
exterieur
то в режиме STRICT объект не исключается, а помечается как:
MAYBE_TERRAIN
причина: terrain_missing
Что это значит по смыслу
Нет подтверждения наличия внешнего пространства.
Но это не считается жёстким стоп-фактором на этапе текстового отбора.

3. Когда объявление становится ✅ полностью подходящим
Объявление получает OK, только если:
нет ни одной причины для исключения
нет ни одного недостающего критически важного подтверждения для режима STRICT
не сработал ни один статус MAYBE_*
Проще говоря, OK — это когда бот смог подтвердить всё основное, что ему нужно на текущем этапе.
Фактически для OK должно выполняться следующее
Не должно быть:
аренды
квартиры
коммерции
запрещённых форматов
запрещённого города
площади ниже 90 м²
менее 3 спален
И одновременно должно быть подтверждено либо не вызывать сомнений:
город — из допустимого списка
площадь — указана и не меньше 90
спальни — указаны и не меньше 3
желательно подтверждено внешнее пространство, иначе будет MAYBE_TERRAIN вместо OK

🔍 4. Как обеспечивается качество результатов
1. Удаление дублей
Один и тот же объект не показывается несколько раз
2. Фильтрация «мусорных» ссылок
Система определяет:
страница с объявлением
vs
страница со списком объектов
👉 Если это список:
помечается как ⚠️ «нужно проверить»
или пытается извлечь конкретные объявления
3. Попытка «докопаться до источника»
Если ссылка не прямая:
система пытается найти конкретное объявление внутри
4. Приоритизация
Пользователь получает:
только STRICT + OK (самые релевантные)
ограниченное количество (чтобы не перегружать)
